F1-level. Here's what it defines and why it kills circular failures:
The Two-Layer Architecture — same pattern as your F1 search engine. Assembly (the War Room) decides WHAT goes into the prompt from 7 sources. Delivery (the Transmission) decides HOW it physically reaches the runtime through 4 channels. These are completely separate lifecycles, just like your Data Preparation vs Query Execution layers.
The 7 Injection Sources — every prompt is composed from exactly these, in this order: Identity (who the agent IS), Constraint (what it CANNOT do), Spine State (what the project IS right now — fetched fresh every time, never cached, 50ms cost), Skill (domain expertise matched by trigger-map), Mandate (the specific task from the DAG step), Handoff (compressed previous step output from PA), and Correction (the mutation engine — only on retry). Total overhead: ~4,700 tokens max, leaving 95% of the effective budget for actual work.
The Mutation Engine — this is the one-shot kill mechanism. Every other system retries with the same prompt and loops forever. Ours builds a Correction block from: the exact error (stderr, exit code), what Bodyguard caught (which checks passed/failed), what Spine shows NOW (filesystem delta), and what approach was already tried. The assembled prompt is SHA-256 hashed — if two consecutive attempts produce the same hash, the system HALTS because identical prompts guarantee identical failures. This is structurally impossible to loop.
The 4 Delivery Channels — CLI stdin (Codex, Claude, Gemini — unlimited size, binary-safe, no shell escaping issues), DOM paste (ChatGPT — ClipboardEvent into contentEditable, 8000 char limit, two-message split for larger payloads), API body (future direct calls), and Session resume (persistent Conductor — only injects spine + mandate + correction, NOT re-injecting identity/skill).
The Routing Table — every agent type is mapped to exactly which sources it receives and through which channel. Reviewers get NO handoff (they evaluate cold). The Debugger gets correction as its PRIMARY input. The Deployer gets env vars via process.env, never in prompt text. Image-gen gets minimal injection — just identity + mandate through DOM.
The Error Capture Pipeline — 7 steps, all code, no LLM. Capture stdout/stderr → snapshot Spine delta → run Bodyguard even on failures → classify error type via regex (CODE_ERROR, AUTH_EXPIRED, RATE_LIMITED, TIMEOUT, DISK_FULL, etc.) → route to specific recovery action. Each error type has a hardcoded recovery path. Auth errors don't mutate the prompt (same prompt, new token). Rate limits defer and resume. Code errors trigger the Mutation Engine.
The Retraction System — when Step 5 discovers Step 3's output was wrong, the system git-reverts Step 3's commits, re-plans from Step 3 onward, and spawns a new Step 3 with DOUBLE correction (original failure context + debugger's root cause analysis). This handles the "passed bodyguard but was still wrong" case.
6 complete scenarios with exact injection routes: happy path, step failure with mutation recovery, rate limit deferral, auth expiry, retraction, and Context Warden kill/respawn.
The key philosophical difference from your F1 doc that maps directly: F1 has 5 search strategies racing in parallel with RRF fusion. This system has 7 injection sources assembled in pipeline with token budget fusion. F1 never lets a query hit the database unprepared. This system never lets an agent start thinking unprepared.
